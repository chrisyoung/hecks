# Hexagonal Architecture Trade-offs in Hecks

Three intentional decisions in Hecks deviate from textbook hexagonal architecture. Each is explained here — what was chosen, why, and what the cost is.

---

## 1. Web Explorer — Intentional Hexagonal Impurity

The Web Explorer (`hecksties/lib/hecks/extensions/web_explorer.rb`) is infrastructure code with deep domain knowledge. It inspects the domain IR at runtime to build forms, render aggregate state, display lifecycle badges, and populate reference dropdowns. This deliberately violates the hexagonal principle that infrastructure should not know about domain internals.

**Why it's justified:**

The Web Explorer lives inside `hecksties/`, the extension layer. It's never shipped as part of the generated domain gem — it runs only in development and exploration contexts. Its purpose is to reflect the domain faithfully, which requires knowing its structure. A generic UI with no domain knowledge can't render typed command forms, lifecycle transitions, or cross-aggregate reference dropdowns.

**Contrast with static build targets:**

Static targets (Ruby, Go, Rails) bake routes and views into the generated output at build time. The relationship between domain and UI is resolved once, at generation time, with no runtime coupling. The Web Explorer takes the opposite approach: it reads the IR at every request, which is what makes it universal.

**How it's contained:**

```ruby
Hecks.describe_extension(:web_explorer,
  description: "Domain web explorer UI",
  config: {},
  wires_to: :http)

Hecks.register_extension(:web_explorer) do |domain_mod, domain, runtime|
  views_dir = File.expand_path("web_explorer/views", __dir__)
  domain_mod.instance_variable_set(:@_web_explorer_views, views_dir)
  domain_mod.instance_variable_set(:@_web_explorer_domain, domain)
end
```

The domain knowledge stays inside the extension registration block. It doesn't bleed into the domain module's own code or the generated gem.

**Rule of thumb:** Don't replicate the Web Explorer pattern. If you're writing an extension and it reaches into the domain IR, ask whether it belongs in `hecksties/` as a development-only tool or whether it should be a static generator instead.

## 2. Ports vs Gates — Naming and Responsibility Split

Hecks uses two terms that look related but serve entirely different roles: **ports** and **gates**.

### Ports — repository interfaces

"Ports" in Hecks refers to the repository interface: the set of persistence methods (`find`, `all`, `save`, `delete`, etc.) that the runtime binds to aggregate classes via `PortSetup#wire_aggregate`. The `PortGenerator` in the Bluebook generates a `Ports::PizzaRepository` stub module with `NotImplementedError` placeholders.

This matches the hexagonal architecture meaning of "port" — an interface boundary between the domain and an adapter.

### Gates — access control roles

"Gates" are role-based access restrictions declared in the Hecksagon (infrastructure wiring). A gate names a role and the methods that role is allowed to call. `GateEnforcer` applies those restrictions at runtime by redefining disallowed methods to raise `Hecks::GateAccessDenied`.

```ruby
# In the Hecksagon (hecks_hecksagon.rb):
gate "Pizza", :admin do
  allow :find, :all, :create_pizza, :add_topping
end

gate "Pizza", :guest do
  allow :find, :all
end
```

### Why the split?

Access control depends on deployment context. The same domain might expose a public API (guest gate), an admin panel (admin gate), and an internal service-to-service interface (no gate at all). Putting access control in the domain DSL couples the domain to a specific deployment topology. Gates belong in the Hecksagon because that's where infrastructure decisions live.

The Bluebook `port` keyword still exists for backward compatibility but is now a no-op:

```ruby
# constraint_methods.rb — kept as no-op for compatibility
def port(_name, _methods = nil, &_block); end
```

Existing `port` declarations in domain files are silently ignored. Migrate them to `gate` declarations in the Hecksagon.

**Summary:**

| Term | Where declared | What it does |
|------|----------------|--------------|
| Port | Generated by `PortGenerator`; methods wired by `PortSetup` | Repository interface between domain and persistence adapter |
| Gate | Hecksagon DSL; enforced by `GateEnforcer` | Role-based access control for aggregate methods |

See the [Hecksagon DSL Reference](hecksagon_reference.md) for the full `gate` syntax.

## 3. Implicit Application Service Layer

Hecks has no `ApplicationService` class. The `Runtime` is the application service.

### What wires everything together

`PortSetup#wire_aggregate` is the orchestration point. For each aggregate, it binds five concerns in sequence:

```ruby
def wire_aggregate(agg)
  agg_class = @mod.const_get(domain_constant_name(agg.name))
  repo      = @repositories[agg.name]
  defaults  = build_defaults(agg)

  Persistence.bind(agg_class, agg, repo)             # find, all, save, delete
  Commands.bind(agg_class, agg, @command_bus, repo, defaults) # create, update, commands
  Querying.bind(agg_class, agg)                      # where, first, last, count
  Introspection.bind(agg_class, agg)                 # hecks_attributes, hecks_aggregate
  Versioning.bind(agg_class, repo) if agg.versioned?
  AttachmentMethods.bind(agg_class) if agg.attachable?
  wire_query_objects(agg, agg_class)
  GateEnforcer.new(gate_name: @gate_name, hecksagon: @hecksagon).enforce!(agg, agg_class)
end
```

### CommandBus as the service layer

The `CommandBus` provides what a traditional application service layer would: a place to plug in cross-cutting concerns (logging, validation, idempotency, authorization) without those concerns appearing inside domain objects.

```ruby
bus = CommandBus.new(domain: domain, event_bus: event_bus)

bus.use :logging do |command, next_handler|
  puts "Dispatching #{command.class.name}"
  result = next_handler.call
  puts "Done"
  result
end

bus.dispatch("CreatePizza", name: "Margherita")
# => #<PizzasDomain::Pizza::Events::CreatedPizza>
```

Middleware wraps inward, Rack-style. The innermost handler creates the domain event and publishes it. Middleware sees the command before and after, but never touches aggregate internals.

### Trade-offs

**Benefit:** Zero boilerplate. `Pizza.create(name: "Margherita")` runs the full pipeline — validation, persistence, event publication, middleware — without an application service class to write or maintain.

**Cost:** The execution path is harder to follow for newcomers. When `Pizza.create` is called, control passes from `Commands.bind`'s dynamically defined singleton method through the `CommandBus` middleware chain to the persistence adapter, then back out through the event bus. No single file shows this flow. Start with `PortSetup#wire_aggregate`.

**Where to look:**
- `hecksties/lib/hecks/runtime/port_setup.rb` — wiring orchestration
- `hecksties/lib/hecks/ports/commands/command_bus.rb` — middleware pipeline
- `hecksties/lib/hecks/ports/commands/` — individual command binding strategies
